# Codex Atelier Cross-Check Guide

## 何时启用交叉检查

出现以下任一条件时，默认启用交叉检查：

- 修改核心业务逻辑或高风险路径
- 任务涉及 3 个以上关键文件
- 需要跨模块、跨层联动
- 需求或根因不够明确
- 用户明确要求更稳的质量控制

## 角色分离

| 角色 | 目标 | 写权限 |
| --- | --- | --- |
| Executor | 实现或修复 | `workspace-write` |
| Reviewer | 审查实现质量和回归风险 | `read-only` |
| Verifier | 独立确认验收 | `read-only` |
| Investigator | 处理冲突结论 | `read-only` |

原则：

- Executor 不做最终验收
- Reviewer 不复写实现
- Verifier 不继承“已通过”结论
- Investigator 只处理差异本身，不重做整包任务

## 四种交叉检查模式

### Mode A: Review

适用于实现路径明确，但需要独立 code review 的任务。

流程：

1. Executor 完成实现
2. Claude 整理变更摘要和关键文件
3. Reviewer 独立审查 bug、回归、遗漏测试、边界处理
4. Claude 根据审查结论决定通过、修复或再审

### Mode B: Verify

适用于需要独立运行命令或独立复现问题的任务。

流程：

1. Claude 提供验收标准与关键命令
2. Verifier 独立运行命令或复现故障
3. Verifier 返回通过、失败或无法执行及其证据
4. Claude 依据证据收口

### Mode C: Compare

适用于实现方案不唯一、希望择优的任务。

流程：

1. 将同一目标拆成两个不同提示词或两个不同实现路径
2. 并行派发给两个 Executor
3. Claude 对比正确性、复杂度、可维护性、边界处理
4. 选择更优实现，必要时再发微调指令

### Mode D: Dispute

适用于 Executor、Reviewer、Verifier 结论冲突的任务。

流程：

1. Claude 整理争议点
2. 列出双方证据
3. 让 Investigator 只判断差异点
4. 按最小验证动作继续，而不是整包返工

## 冲突表模板

| 字段 | 内容 |
| --- | --- |
| 争议点 | 哪个结论不一致 |
| 证据 A | 一侧的文件、命令、日志 |
| 证据 B | 另一侧的文件、命令、日志 |
| 待裁决问题 | 需要再次验证的最小问题 |
| 下一跳角色 | 最适合继续处理的人 |

## 质量门

- 没有明确验收标准，不收口
- 没有独立证据，不算完成验证
- Reviewer 没有独立发现，不算完成评审
- Verifier 不能执行时，要明确说明阻塞原因
- 冲突存在时，Claude 不能跳过仲裁直接宣称完成

## 常见错误

- 让同一个 Codex 同时做实现、评审、验证
- 把多个 Codex 派到相同写入范围
- 因为排期紧就省略独立验证
- 用“看起来没问题”代替命令结果和证据
